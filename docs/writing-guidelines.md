# 写作细则

这份文档用于补充 `AGENTS.md` 中的写作规则。凡是新建、改写或发布博客文章，都应先参考这里的流程；需要完整 workflow 时使用 `.codex/skills/blog-writing-publish-workflow/SKILL.md`。

## 协作方式

- 根据任务选择协作粒度。结构和方向清楚时，可以直接做整篇起草、整篇翻译或整篇改稿；只有主题、提纲、语气还在探索时，才按小节推进。
- 先确认主题、读者和主要方向；完成整篇或关键块后，再集中吸收用户反馈并继续打磨。
- 用户提出新的长期写作偏好或发布习惯时，先询问是否写入 `AGENTS.md` 或本文件。
- 如果用户只是在本次文章里做临时取舍，不要自动沉淀为长期规范。

## 文章定位

- 默认写成带个人观察的技术复盘，而不是泛教程、新闻摘要、产品介绍或资料整理报告。
- 开头尽快交代真实问题：发生了什么、为什么值得记录、这次复盘想保留下什么判断。
- 不使用“为什么这篇文章值得复盘”这类固定章节标题；开头默认用一个工程或生活例子导入，再自然引出文章判断。
- 文章主体优先回答：为什么这样做、遇到了什么取舍、哪里出错、最后如何验证。
- 阅读笔记默认照来源结构写：保留来源标题层级、段落顺序和论证推进；可以逐段翻译、逐段转述或翻译后评论，但不要自行重排成新的文章结构。
- 来源链接已经放在文章开头时，正文不要反复写“原文提到”“作者强调”“文章指出”。直接写内容；只有在区分不同观点、保留关键归因或说明争议时，才显式点出来源。
- 当正文出现两种及以上对象、方案、阶段或取舍的对比时，优先考虑用表格呈现；表格只用于压缩清晰差异，不为装饰而添加。
- 结尾应沉淀可复用经验，而不是简单重复正文。

## 结构流程

- 先提出 1 个候选标题方向和文章结构，等待用户确认。基于外部文章的阅读笔记，文章结构应直接沿用来源结构；没有标题的来源按段落顺序分组，不另起一套 3 到 6 节框架。
- 结构未定时，每个小节先讨论要点，再写正文；结构已定时，可以直接做整篇通稿。
- 小节之间要有因果或排查顺序，避免变成无关联笔记拼接。
- 如果文章来自外部材料，保留来源链接用于回查，并把外部观点和本项目经验区分清楚。正文不强制保留英文原句、原始引文或逐字引用。
- 给出来源后，正文默认进入自己的叙述，不用持续用“原文/作者/文章”作为段落主语。

## 审稿流程

- 长文草稿默认先保持 `draft: true`，再进入 Proof 网页审稿；不要为了审稿提前发布或生成公开路由。
- Proof 默认用于成稿后的整篇通读和集中批注；结构确认和早期语气校准仍优先在聊天和本地 Markdown 中完成。
- 使用 `npm run proof:review -- <slug-or-repo-relative-post-path>` 把本地草稿上传到 Proof，并把返回的 Proof URL 发给审稿人。
- 审稿意见优先留在 Proof 的评论线程里；agent 读取评论后，在 Proof 线程中回复、处理可接受修改，再把审过的 Markdown 同步回 `src/content/posts/`。
- Proof URL、access token、owner secret 和审稿状态只在当次会话或浏览器里使用，不写入仓库文档、commit message 或文章正文。
- Proof 适合正文阅读、批注和改稿；最终发布前仍要按本仓库流程运行 `npm run check` 和 `npm run build`，涉及路由或渲染时还要运行 `npm test`。

## 语言风格

- 默认使用中文。
- 避免第一人称 `我`，优先使用 `笔者` 或客观句式。
- 不写与主题无关的元说明，例如“这不是直译”“本文为中文读者改写”“由 AI 生成”。
- 少用空泛形容词，多写具体命令、报错、配置、路径、失败现象和决策依据。
- 允许有个人判断，但判断要落在具体事实上。
- 基于外部材料改写或翻译时，最后必须去掉翻译腔：不要保留英文语序、硬直译连接词或不自然的被动句；优先写成自然、顺畅、能直接发布的中文。

## 声音规则

- 文章应像技术作者把一个问题想明白后写下的随笔，而不是把资料重新排版成报告。
- 作者可以在场，但不要频繁自称。优先通过选择的例子、取舍顺序和判断力度体现个人视角。
- 少追求段落工整。允许有短段、转折段和具体观察，让文章读起来像沿着问题往前走。
- 完整覆盖不等于机械照搬。来源里的每个主要段落都要有中文去向，且顺序默认不变；表格、补充解释和个人判断应放在对应来源段落附近，不把材料重新组织成另一篇文章。
- 允许全文翻译作为写作形态；默认仍优先做全覆盖中文转述或节选翻译，避免整篇逐字搬运。
- 翻译或转述后的每段都要再读一遍，检查是否像中文作者自然写出来的句子；如果只是“英文句子换成中文词”，要重写。
- 每个抽象判断后面都要能接上一个动作、现象或失败方式；接不上时，先改成更具体的句子。
- 结尾不要写主持人式总结。文章可以停在一个具体边界、一次取舍或一个仍需保留的警惕上。

## Frontmatter

- 新文章默认带 `draft: true`。
- `summary` 要短，并且能用于列表页和 RSS；不要写成空泛的“本文介绍……”。
- `tags` 应反映文章真实主题，数量保持克制。
- 日期使用项目已有文章的日期格式。

## 发布前检查

- 未经用户明确批准，不移除 `draft: true`，也不创建发布 PR。
- 用户批准发布后，再移除 `draft: true` 或设置 `draft: false`。
- 内容变更优先运行 `npm run check` 和 `npm run build`。
- 若改动涉及路由、布局、组件、内容处理或测试覆盖范围，还应运行 `npm test`。
- 每次发布通过 pull request 准备，不直接推送到 `main`，除非用户明确要求。
- 发布 PR 检查通过且 scope 干净时，Codex 可以自行合并 PR，不再要求用户手动审 PR。

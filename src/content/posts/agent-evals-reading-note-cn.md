很多 Agent 项目早期都能靠手工试用推进：改一段 prompt，跑几个熟悉案例，看看输出有没有明显变好。这个阶段的反馈很快，也很有成就感。但系统一旦进入真实用户场景，问题会变得尴尬：用户说“最近好像变差了”，研发团队却很难判断到底是模型退步、prompt 改坏、工具链出错，还是某个偶发案例被放大了。

Anthropic 这篇文章的价值，不在于列了一组评测框架名称，而是把 Agent eval 放回工程现场：评测要回答的不是“这个模型厉不厉害”，而是“这个 Agent 在具体任务、具体环境、具体约束下，是否稳定地产生了用户真正需要的结果”。换句话说，eval 要把模糊的体感问题，变成可以复现、可以归因、可以讨论的工程信号。

先把几个容易混在一起的词放在同一张桌面上：

| 术语 | 可以先这样理解 |
| --- | --- |
| `task` | 一道题，包含输入和成功标准。 |
| `trial` | 对同一道题的一次尝试。因为 Agent 有随机性，同一个 task 往往要跑多次 trial。 |
| `transcript` | 一次 trial 的完整过程记录，包括模型回复、工具调用、中间结果和交互历史。 |
| `outcome` | trial 结束后环境里的最终事实，比如数据库、文件系统、页面状态、测试结果。 |
| `grader` | 判分逻辑，可以是单元测试、状态检查、静态分析、LLM judge 或人工审核。 |
| `rubric` | 评分细则，提前写清楚按哪些维度判断好坏，比如是否基于政策、是否解释清楚。 |
| `agent harness` | 让模型像 Agent 一样行动的运行框架，负责组织输入、工具调用和返回结果。 |
| `evaluation harness` | 跑评测的基础设施，负责任务调度、环境隔离、记录过程、调用 grader 和汇总结果。 |
| `suite` | 一组围绕同类能力或行为设计的 task。 |

文中的 `reservation` 不是 eval 通用术语，只是订票例子里的“预订记录”。它用来说明一个关键区别：Agent 说“已经订好了”属于 transcript，数据库里真的出现一条预订记录才属于 outcome。

## 为什么 Agent 更难评测

普通 LLM 评测通常比较直观：给一个 prompt，拿到一个 response，再用规则、参考答案或人工判断打分。这里的对象相对稳定，评测逻辑也容易围绕最终文本展开。Agent 不一样，它不是只生成一段答案，而是在多轮交互里调用工具、读取环境、修改状态，并根据中间结果不断调整下一步动作。

这带来第一个麻烦：错误会传播。单轮问答里，一个不准确的句子通常只影响这次输出；Agent 如果在第 2 轮读错文件、查错数据库、点错页面，后面的工具调用可能都会建立在错误状态上。最终失败并不一定来自最后一步，而可能来自很早之前某个不起眼的动作。

第二个麻烦是，最终文本不等于最终结果。一个订票 Agent 可以在对话末尾说“航班已经预订成功”，但真正该检查的是数据库里有没有生成预订记录；一个代码 Agent 可以解释得很像修好了 bug，但真正该检查的是测试是否通过、漏洞是否被堵住、旧行为是否没有被破坏。Anthropic 用 `outcome` 这个词强调的正是这一点：评测不能只看 `transcript` 里说了什么，还要看环境最后变成了什么。

第三个麻烦是，Agent 可能用评测者没预料到的路径完成任务。传统测试容易假设“正确做法”只有一种，但强模型会绕开静态脚本的边界。原文提到 Claude Opus 4.5 在一个航班预订基准里发现了政策漏洞，按原评测标准算失败，但从用户目标看反而可能是更好的解法。这个例子提醒评测设计者：如果 grader 只奖励固定路径，就可能惩罚真正有效的 Agent 行为。

所以，Agent eval 不能退化成“输出像不像参考答案”。更合理的做法是分层看：任务输入是否清楚，环境是否隔离，工具调用是否留下完整记录，最终状态是否符合目标，grader 是否允许合理的替代解法。分清这些层之后，eval 才能解释“为什么失败”，而不只是给出一个冷冰冰的分数。

## Eval 里真正要分清的对象

读 Agent eval 时最容易混乱的一点，是把“题目”“运行过程”“最终结果”“判分逻辑”全部叫成评测。这样说在日常讨论里没问题，但一旦要排查失败，就会变得很不够用。分数下降时，需要知道下降的是哪一层：任务写得含糊，Agent 执行不稳定，环境没有重置干净，还是 grader 判错了。

`task` 是最小评测单元。它不只是给 Agent 的一句指令，还应该包含隐含或显式的成功标准。比如“帮用户改签到明天上午的航班”就是一个 task，但如果没有写清预算、航空公司政策、是否允许转机、最终需要检查什么状态，这道题就很容易变成各说各话。好的 task 应该让两个熟悉业务的人独立判断时，基本能得出同样的通过或失败结论。

`trial` 是同一道 task 的一次运行。这个概念很重要，因为 Agent 不是确定性程序：同一条指令、同一套工具、同一个模型，也可能因为采样、上下文、外部环境或工具返回差异，跑出不同路径。一次通过不能证明稳定，一次失败也不一定代表能力完全缺失。把 task 和 trial 分开，才谈得上后面的 pass@k、pass^k，以及“这个 Agent 到底是偶尔会做对，还是每次都能做对”。

`transcript` 是 trial 的过程记录。它像一次调试日志：模型说了什么、调用了什么工具、工具返回了什么、哪一步改变了状态、后面又如何基于这个中间结果继续行动。没有 transcript，评测失败只能看到一个红叉；有了 transcript，才能判断 Agent 是没理解任务、用了错误工具、误读工具结果，还是其实做对了但被 grader 误判。

`outcome` 是 trial 结束后环境里的事实。订票场景里，outcome 是数据库是否真的有预订记录；代码场景里，outcome 是补丁是否存在、测试是否通过、漏洞是否被修掉；网页操作场景里，outcome 可能是 URL、DOM 状态、后端订单状态或文件是否被正确生成。Anthropic 反复强调 outcome，是因为 Agent 很会“说得像完成了”，但 eval 应该检查“世界是否真的被正确改变了”。

`grader` 才是把 transcript 或 outcome 转成结果的判分逻辑。代码 Agent 可以用单元测试、静态分析、状态检查；客服 Agent 可能要用数据库状态、轮数限制和 LLM rubric，也就是让模型按一份评分细则判断对话质量；研究 Agent 可能要看来源是否可靠、关键事实是否覆盖、结论是否被证据支撑。grader 不必只有一个，复杂 task 往往需要多个 grader 组合，甚至给部分完成的行为一些部分分。

这里还要分清两种 harness。`agent harness` 是被评测对象的一部分：它负责把模型、工具、系统提示词、消息历史和用户输入组织成一个能行动的 Agent。换句话说，评测“某个 Agent”时，测到的不是纯模型能力，而是模型和这层运行框架共同表现出来的能力。

`evaluation harness` 则是评测基础设施：它把 task 发给 Agent，准备干净环境，并发运行多个 trial，保存 transcript，调用 grader，最后汇总结果。很多看似 Agent 变差的问题，其实出在 evaluation harness：环境没隔离、缓存污染、测试依赖外部服务、并发资源不足，都会让评测结果看起来像能力波动。

最后，`suite` 是一组 task 的集合。它不是随便堆题库，而是围绕某类能力或风险组织起来的样本集。比如客服 Agent 可以有退款、取消、升级人工、身份验证几组 suite；代码 Agent 可以有 bug fix、安全修复、重构、测试补全几组 suite。这样做的好处是，当总分变化时，团队能继续往下追问：到底是整体退步，还是某一类能力被新改动伤到了。

把这些对象分开之后，eval 才开始像工程系统。一次失败不再只是“模型没做好”，而可以被拆成更具体的问题：task 是否可解，trial 是否稳定，transcript 是否暴露了错误路径，outcome 是否真的不符合目标，grader 是否公平，harness 是否给了 Agent 合理环境。

## 不要只看最终答案，也不要迷信轨迹

如果只能选一个评测对象，`outcome` 通常比最终回答更接近用户价值。用户要的是机票真的订上、退款真的处理、代码真的修好、报告里的结论真的有依据，而不是 Agent 在最后一句话里宣布“任务完成”。这也是为什么代码 Agent 的评测天然喜欢单元测试和端到端测试：它们检查的是结果，而不是解释。

但只看 outcome 也不够。一个 task 失败时，团队还需要知道失败发生在哪里。Agent 是一开始就误解了用户目标，还是选错工具，还是工具返回了错误但它没有发现，还是最后一步忘了提交状态？这些问题只能从 `transcript` 里看出来。transcript 不一定是最终判分的主要依据，但它是调试和归因的入口。

可以把两者分工理解成这样：outcome 回答“有没有把事情做成”，transcript 回答“为什么做成或没做成”。前者决定这次 trial 对用户有没有价值，后者决定团队下一步应该改 prompt、改工具、改 agent harness，还是修 eval 本身。

真正麻烦的是第三类东西：对轨迹的硬性约束。比如要求 Agent 必须先调用 `search_flights`，再调用 `select_flight`，最后调用 `book_flight`，中间不能走别的路径。这种检查看起来很精确，但很容易把 eval 变脆。Agent 可能先读政策再查航班，或者先确认用户偏好再选航班，这些路径未必错误。如果 grader 只承认一种调用顺序，就会把有效解法判成失败。

这不等于完全不看轨迹。某些路径约束确实有业务意义：处理退款前必须验证身份，转账前必须二次确认，高风险代码修改后必须运行测试，医疗或法律场景不能编造来源。这类约束不是“评测者偏好的实现步骤”，而是产品和安全边界本身。它们应该被写成必要条件。

更稳的做法是：把 outcome 放在主位置，把 transcript 用来解释失败和补充少量关键约束。对于代码 Agent，可以先用测试确认漏洞是否修好，再用 transcript 看它有没有读相关文件、有没有运行测试、有没有因为误解报错而绕远。对于客服 Agent，可以先看工单和退款状态，再看对话里是否解释清楚、是否基于政策工具回答、是否在用户生气时保持合适语气。

这样设计出来的 eval 不会只奖励“像标准答案一样行动”的 Agent，而是奖励“在约束内真正解决问题”的 Agent。Anthropic 文章里反复出现的一个隐含判断是：Agent 越强，越可能找到评测作者没预料到的路线；好的 eval 应该抓住目标和边界，而不是把某条演示路径误当成唯一正确答案。

## 不同 Agent 类型对应不同评测组合

不同 Agent 的难点不一样，eval 也不应该长成同一种样子。比较实用的判断方式是：先问这个 Agent 主要改变什么世界状态，再问哪些过程约束真的重要。

| Agent 类型 | 更该检查的 outcome | transcript 里值得看的东西 |
| --- | --- | --- |
| 代码 Agent | 测试是否通过、漏洞是否修复、旧行为是否没坏。 | 是否读了相关文件、是否运行测试、是否引入过度设计。 |
| 对话 Agent | 工单、订单、退款、预约等业务状态是否正确。 | 是否验证身份、是否基于政策工具回答、语气和轮数是否合适。 |
| 研究 Agent | 结论是否有来源支撑，关键事实是否覆盖。 | 搜索路径、引用来源质量、是否把不确定内容说成确定。 |
| 电脑操作 Agent | 页面、文件、应用配置、后端数据是否变成目标状态。 | 是否选对交互方式，是否误点、漏确认或卡在界面里。 |

代码 Agent 的评测最接近传统软件工程。只要 task 写得清楚，最有力的 grader 往往还是确定性测试：单元测试、集成测试、端到端测试、静态分析、安全扫描。一个修认证绕过漏洞的 Agent，不应该靠解释“已经修好了认证逻辑”来通过，而应该靠测试证明空密码、空值、旧 token 等边界都被正确拒绝。transcript 在这里主要用来补充质量判断，比如是否运行了相关测试，是否为了修一个小 bug 改出一套大框架。

对话 Agent 的评测难在“完成任务”和“交互质量”都重要。客服 Agent 不能只看最后有没有处理退款，还要看有没有验证身份、有没有解释政策、有没有在用户生气时保持合适语气。这里的 outcome 可以是工单状态、退款状态、确认邮件是否发送；transcript 则负责检查对话过程是否合规。原文还提到，这类 eval 经常需要另一个 LLM 扮演用户，因为真实对话不是一次 prompt 就能覆盖的。

研究 Agent 的评测更难，因为很多任务没有唯一标准答案。市场调研、竞品分析、论文综述、尽调报告，正确性往往取决于问题边界和证据质量。一个实用组合是：用来源检查确认关键结论有引用支撑，用覆盖检查确认重要事实没有漏掉，用 LLM rubric 或专家审核判断结构、取舍和综合质量。这里最怕的是“写得很像研究，但证据跟不上结论”，所以 groundedness 应该比文风更靠前。

电脑操作 Agent 的评测重点是环境状态。它通过截图、鼠标、键盘、滚动去操作真实软件，所以最终结果可能藏在浏览器页面、桌面应用、文件系统、数据库或配置文件里。比如买东西不能只看页面出现“确认成功”，还要看后端订单是否真的创建；编辑文档不能只看 Agent 说保存了，还要检查文件内容。对这类 Agent，评测环境的隔离尤其重要，因为登录态、缓存、窗口位置、已有文件都会影响下一次 trial。

还有一层容易被忽略：不同 Agent 对成本和延迟的敏感点也不同。浏览器 Agent 有时读 DOM 更快，但 DOM 太大时会浪费 token；有时看截图更省上下文，但交互更慢。代码 Agent 运行测试可能成本高，但不运行测试又很难建立信任。研究 Agent 可以多搜索几轮提高覆盖，但也更容易引入噪声。eval 如果只看是否成功，不记录 token、工具调用次数、耗时和失败类型，就很难指导真实产品取舍。

所以，按 Agent 类型设计 eval，不是为了做分类学，而是为了避免拿错尺子。代码 Agent 要优先用测试、编译或打包结果、静态检查来确认结果；对话 Agent 要同时满足业务状态和交互质量；研究 Agent 要把证据链放在漂亮表达之前；电脑操作 Agent 要检查真实软件环境是否被正确改变。评测组合越贴近 Agent 真正承担的工作，分数才越像工程信号。

## 从 0 到 1 搭 Eval 的现实路径

一套 eval 不需要从几百道题开始。原文给的建议很实在：早期先从 20 到 50 个简单 task 起步，而且最好来自真实失败。真实失败包括用户报过的问题、开发时手工复现过的坏例子、发布前每次都要人工点一遍的流程、团队成员反复争论的边界情况。这些案例不一定覆盖全部能力，但能立刻把“感觉不稳定”变成可回归的问题集。

第一步是收集现在已经在手工测的东西。比如每次改客服 Agent，都有人手动试“用户因退款问题很不满，Agent 需要按政策处理”“用户没验证身份就要求退款”“政策工具返回不支持退款”这几类场景；每次改代码 Agent，都有人试“修一个明确 bug”“不要改无关文件”“修完要跑测试”。这些手工检查本来就是隐形 eval，只是还没有被写成可重复运行的 task。

第二步是把 task 写得足够明确。一个好的 task 应该让 Agent 知道要做什么，也让 grader 知道该检查什么。原文里有个很典型的坑：任务要求 Agent 写一个脚本，但没有指定文件路径，测试却默认脚本必须出现在某个路径下。这样 Agent 失败不一定是能力问题，而是题目没把隐藏要求说出来。eval 里凡是 grader 会检查的东西，都应该尽量在 task 里讲清楚。

第三步是给每个 task 准备参考解。参考解不是让 Agent 照抄的标准答案，而是一个已知能通过 grader 的人工或基准实现。它有两个作用：证明这道题确实可解，也证明 grader 没有配置错。如果一个强模型跑很多次都是 0% 通过，先不要急着说模型不行，应该先怀疑 task 是否含糊、环境是否坏了、grader 是否把正确结果判错了。

第四步是补正反例。正例是“该触发某种行为”的场景，反例是“不该触发”的场景。只测正例会把 Agent 训练成过度触发。比如只测试“什么时候应该搜索网页”，可能得到一个什么问题都先搜索的 Agent；只测试“什么时候应该调用退款工具”，可能让 Agent 在没有验证身份时也急着处理退款。正反例一起放进 suite，才能同时压住漏做和乱做。

第五步是保证每次 trial 从干净环境开始。这里的干净不是抽象词，而是数据库没有前一次留下的订单，文件系统没有旧补丁和临时文件，浏览器没有会影响任务的登录态或打开页面，服务没有残留缓存或限流状态。否则一次 trial 的通过或失败，可能测到的不是 Agent 能力，而是上一轮运行留下的东西。

第六步是先用确定性 grader，再补模型或人工判断。能用测试、状态检查、静态分析解决的，就先不要交给 LLM judge；开放式质量、语气、综合判断这类很难写成规则的，再用 rubric 让模型评。即使用 LLM judge，也要经常抽样让人看 transcript 和判分结果，确认它没有把有依据的回答判错，也没有被流畅但空洞的回答骗过。

最后，grader 要允许合理的部分成功。一个客服 Agent 正确识别问题、验证身份、解释政策，但最后退款接口失败，和一个一开始就答非所问的 Agent，不应该拿到同样的分数。部分分不是为了宽容错误，而是为了让分数能告诉团队“系统卡在哪一段”。如果只有全对或全错，很多改进信号会被压成同一个红叉。

从 0 到 1 的 eval 更像把团队已经知道的失败写进版本库。先不追求完美覆盖，而是让每次修 bug、改 prompt、换模型后，都能跑一组真实、明确、可解释的 task。只要这套小 eval 能持续接住回归，它就已经开始产生工程价值。

## 长期维护：读 transcript，比盯分数更重要

eval 建起来之后，最容易犯的错误是只盯总分。分数当然重要，但它只是入口。真正能让团队进步的，是定期读失败 trial 的 transcript，确认失败是否“公平”：Agent 到底哪里做错了，grader 有没有误判，task 有没有隐藏条件，环境有没有让本来独立的 trial 互相影响。

这件事听起来笨，但很关键。一个 eval 如果不读 transcript，很容易产生两种错觉：分数低时，以为 Agent 不行，其实是题目写坏了；分数高时，以为系统稳定，其实是 grader 太宽、task 太简单，或者 suite 没覆盖真实用户会遇到的新失败。Anthropic 原文里反复强调，评测分数不能脱离 transcript 和具体任务细节单独解释。

还要分清两类 suite。第一类是能力评测：里面放的是 Agent 现在还不太会、但希望未来会做的题，初始通过率可以比较低。它的作用是告诉团队下一座山在哪里。第二类是回归评测：里面放的是 Agent 已经应该稳定做好的题，通过率应该接近 100%。它的作用是防止一次 prompt、工具或模型升级，把旧能力改坏。

当一组能力评测通过率越来越高，它就可能逐渐变成回归评测。反过来，如果一组 task 长期接近满分，就要小心它已经太简单，不能再说明能力增长。原文把这叫 eval saturation，也就是评测饱和：题目已经区分不出新模型或新 Agent 的真实进步。这时继续看总分意义不大，应该补更难、更贴近真实任务的新题。

非确定性也要长期记录。同一个 task 跑一次通过，不代表它稳定；跑一次失败，也不代表它完全不会。对有些场景，团队关心的是“多试几次至少有一次成功”，比如生成多个候选方案；对客服、订票、支付这类用户直接面对的场景，团队更关心“每次都要成功”。所以除了平均分，还应该记录同一 task 多次 trial 的波动，避免被单次结果带偏。

维护 eval 的另一个来源是真实世界的新失败。用户反馈、线上监控、人工复查、A/B 测试里发现的问题，都应该定期回流成 task。这样 suite 才不会停留在项目早期的想象里。一个健康的 eval 集合，应该像测试用例一样被持续修订：新失败进来，过时任务删掉，含糊题重写，grader 判错就修 grader。

最后，eval 也可以反过来定义产品需求。与其说“希望 Agent 更聪明一点”，不如先写出几道它现在还做不好的 task：应该查哪些信息、允许调用哪些工具、最终状态是什么、哪些失败不能接受。这样做不是为了追求形式，而是让产品、工程、研究能围绕同一组具体案例讨论。分数只是表面，真正有用的是这些案例把“感觉变差了”变成了可以复现、可以修、可以继续积累的工程语言。

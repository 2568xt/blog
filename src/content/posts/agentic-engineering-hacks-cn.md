## 为什么这篇文章值得复盘

Matt Van Horn 这篇 X Article 表面上是一份很长的技巧清单：`plan.md`、`/ce-plan`、语音输入、并行标签页、`cmux`、权限绕过、`last30days`、原始 transcript、Human Signal、skills、开源贡献、Printing Press，最后还专门谈到 AI psychosis 这样的风险。单独看每一条，都像是某个重度使用者的个人偏好；连在一起看，它其实在回答一个更大的问题：Agentic Engineering 到底怎样从“会用工具”变成“能稳定产出”的工程系统。

这也是它比普通工具列表更值得复盘的地方。很多 agent 工作流的问题，并不是模型完全不能做，而是任务意图没有被写成可执行计划，执行过程没有被拆成可观察的单元，失败信号没有回到原始记录，成功经验也没有沉淀成下一次能复用的规则。于是每次看起来都在“让 AI 帮忙”，但每次都要重新解释、重新纠偏、重新承受同一类风险。

原文真正有价值的判断，是把 agent 当成一个需要工作制度的工程队友，而不是一次性问答界面。计划不是为了让人逐字阅读，而是为了给 agent 一个可执行边界；多标签和并行会话不是为了热闹，而是为了让不同任务线彼此隔离；原始 transcript 和 Human Signal 不是额外记录癖，而是让质量判断回到真实输入、真实反馈和真实失败。

所以这篇复盘不打算把 22 条技巧逐条翻译一遍。更值得保留下来的，是它背后的五层结构：先把意图变成计划，再把执行拆成可并行但可审查的工作单元，随后用原始信号判断质量，把有效做法沉淀成技能和规则，最后持续提醒自己哪些权限、成本和心理风险不能交给自动化惯性处理。真正的门槛不在某一个插件，而在能不能把这些动作连成一套可持续的工作方式。

## 先把意图写成可执行计划

原文把 `plan.md` 放在第一条，不是因为计划文档本身神圣，而是因为它改变了 agent 接收任务的方式。一个“把这个做了”的请求，很容易让 agent 在还没理解约束时就开始行动；一个计划则会迫使它先整理目标、上下文、路径、验收条件和已有模式。对人来说，这像是多了一层手续；对 agent 来说，这是把模糊意图变成可执行边界。

Matt Van Horn 的用法很激进：产品想法、GitHub issue、终端报错、截图、设计稿、Slack 讨论，都可以先丢进 `/ce-plan`；如果想法还不成形，就先用 `/ce-brainstorm` 把问题问清楚，再进入计划。这里的关键不是“换一个命令”，而是把所有输入都当成计划材料。截图不是让模型即兴发挥的装饰，报错也不是让模型猜修法的入口，它们都要被整理成一份能约束后续执行的上下文。

计划阶段还承担了研究工作。原文提到，`/ce-plan` 会并行读取代码库、寻找已有模式、检查项目约定，也会回看过去的解决方案；必要时再查外部文档和最佳实践。这样生成的计划不是泛泛建议，而应该落在当前仓库、当前历史和当前约束里。换句话说，计划的价值不只是“列步骤”，而是先把 agent 可能偷懒或误判的地方暴露出来。

更有意思的是，原文并不鼓励人逐字读完 `plan.md`。计划主要是给 agent 的：它让 agent 写下方法、验收标准和完成边界，然后由 `/ce-work` 接着执行。人类操作者更适合做的是抽查标题、问一句为什么采用这个方案、要求一段 TLDR，或者让 agent 用更简单的话解释计划。这样既不把人拖回 300 行 Markdown 的人工审稿里，也不会完全放弃对方向的判断。

这套做法同样适合非代码工作。原文举了一个很典型的深度知识任务：不要直接要求 agent 写最终文档，而是先让它制定“如何阅读资料、如何消化会议 transcript、如何把业务问题和外部方法论组织成可用产物”的计划。直接要结果时，模型容易给出一份看似完整但浅的产物；先要求它规划生成路径，再按计划执行，反而能逼出更深的阅读、取舍和整合。

所以，`plan.md` 的本质不是文档，而是 checkpoint。上下文爆掉了，可以新开会话继续按计划执行；任务变复杂了，可以回到计划里重新判断边界；执行结果不对，也能对照计划看是理解错、研究不够，还是验收条件写得太松。Agentic Engineering 的第一步，正是先让意图有一个能被传递、审查和恢复的形状。

## 执行层不是单 Agent 蛮跑

如果计划解决的是“做什么”和“怎么算做完”，执行层解决的就是“怎样让任务持续流动”。原文里语音输入、`cmux` 多标签、终端默认进入 Claude 或 Codex、远程控制、AgentMail、权限绕过、Claude 和 Codex 分工，表面上像一堆个人工具偏好；放在一起看，它们都在降低同一件事的成本：把一个新想法变成一个正在运行的 agent 会话。

语音输入对应的是意图带宽。原文的判断很直接：给 LLM 说话不需要像语音转文字那样追求逐字准确，因为另一端能根据上下文修正口误、停顿和半截句子。这件事改变的是输入习惯。键盘容易让人把上下文压缩成一句命令，语音反而允许把背景、犹豫、边界条件一口气倒出来。它不保证内容更严谨，但能让 agent 更早接触到真实问题的形状。

多标签和 `cmux` 解决的是吞吐量，而不是炫耀并发。原文描述的工作台是四到六个独立会话：一个在写计划，一个在按另一个计划执行，一个在跑研究，一个在修刚测试出来的 bug。关键不是让一个 agent 同时承担所有上下文，而是让每条任务线都有自己的边界、状态和等待点。没有计划时，并行会话只会制造混乱；有了计划，每个标签页才像一个可切换的工作单元。

终端默认进入 Claude 或 Codex，则是在压低启动成本。新标签一打开就是 agent，而不是先进入 shell、再切目录、再启动工具。这个做法听起来很极端，但它揭示了一个常被忽略的事实：新会话越便宜，越容易把临时想法、报错、边角任务及时丢给 agent，而不是攒到最后变成一团上下文债。执行系统的效率，有时不是来自更聪明的模型，而是来自更少的启动摩擦。

远程控制和 AgentMail 把这个工作台继续往外扩。原文里，一个桌面会话可以从手机接着控制；一封带主题、正文和附件的邮件，也可以触发新的 Claude Code 会话开始工作。这让 agent 从“电脑前的工具”变成了“随时可投递任务的入口”。但这里不能只看方便：原文也提到 allowlist、DKIM、SPF 这类门槛，说明远程入口一旦打开，就必须同时设计谁能触发、什么输入可信、失败邮件是否应该直接落地成任务。

权限策略也是执行层的一部分。原文对 YOLO 模式的态度很明确：多会话并行时，如果每个命令和编辑都要人工确认，系统会慢到不可用；声音 hook 则用来提示哪个会话刚刚结束。这个判断不等于权限越大越好，而是说明权限模型必须和工作方式匹配。高吞吐执行需要更少打断，但权限越宽，错误影响面也越大。后面谈风险边界时，还要单独回到这个问题。

Claude 和 Codex 的分工，则是把“多个 agent”从口号落到实际路由。原文的个人习惯是 Claude 做计划和品味判断，Codex 承担大量构建；也有人反过来，让 Codex 构建、Claude 复核。这里真正重要的不是哪个模型更适合某个身份，而是任务被拆成了不同职责：谁负责研究和计划，谁负责批量实现，谁负责审查和收口。执行层成熟以后，agent 不再是一个聊天框，而是一组可以互相接力的工作位。

所以，执行层的重点不是“让一个 agent 放开跑”。更稳的理解是：提高输入带宽，降低启动摩擦，把任务拆到多个可恢复会话里，用通知和权限策略控制等待点，再用模型分工承接不同阶段。这样看，原文那些看似零散的工具技巧，其实都在服务同一个目标：让计划真的进入持续执行，而不是停在一份漂亮的 Markdown 里。

## 质量判断要回到原始信号

计划和并行执行能把产量拉起来，但产量一高，另一个问题会立刻出现：怎么判断 agent 做出来的东西是不是对的。原文在中段连续谈 `last30days`、Granola raw transcript、Human Signal 和 HyperFrames，重点并不是又多了几种输入源，而是把质量判断从“模型说得像不像”拉回到真实世界的信号里。

`last30days` 解决的是时间问题。模型训练数据会过期，官方文档也未必能覆盖社区最近踩过的坑。原文举的例子是比较 Vercel agent-browser 和 Playwright：先用 `/last30days` 拉 Reddit、X、YouTube、HN、GitHub 等近期讨论，再把结果喂给 `/ce-plan`。这样计划不是从静态记忆出发，而是把最近三十天的使用反馈、争议和经验纳入判断。对选库、做功能、写文章、见合作伙伴这类任务来说，先研究再计划，比先拍脑袋再补证据稳得多。

Granola 这一条解决的是上下文被过度加工的问题。原文强调要把会议的 raw transcript 直接丢给 agent，而不是先由人总结一遍。原因很简单：总结会提前做取舍，可能把真正有价值的细节、语气、犹豫和旁枝剪掉。原始 transcript 里有噪声，比如吃饭闲聊、重复表达、半句话，但也保留了问题真实发生的形状。Agent 的任务不是照单全收，而是在完整材料里抽取和当前代码库、既有计划、业务问题真正相关的部分。

这和前面的语音输入其实是一条线：越早把原始材料交给 agent，越要保留足够多的上下文，让后续判断有东西可追。一个会议被整理成三条结论，看起来很干净，但失败时很难知道遗漏发生在哪里；一份 raw transcript 虽然脏，却能在需要时回到具体句子、具体分歧和具体承诺。质量控制不是把输入变漂亮，而是让关键事实可追溯。

Human Signal 则把人的角色重新放准。原文说，多 agent 工作流里，人不应该继续把自己当成执行的手，而应该成为信号：看结果，判断哪一个方向更近，指出最大风险，要求改语言、改结构、改重点。这里的“人”不是在每一行里亲自加工，而是在反应和重定向循环里提供 taste、direction 和 judgment。Agent 提供体量，人提供取舍。

这个判断对写作尤其明显。让 agent 产出十个版本并不难，难的是判断哪一版抓住了真正的问题，哪一段虽然流畅但偷换了来源，哪一个比喻把概念讲偏了。质量不来自更多文本，而来自更快地暴露差异、更准确地指出方向。Human Signal 的价值就在这里：把“看起来完成了”拉回“是否解决了真实问题”。

HyperFrames 看似是视频工具，其实也在说明同一个反馈循环可以迁移到更多媒介。原文的做法是用 `script.md`、分镜、字幕、动态排版来让 agent 生成视频，输出从 PR 变成 MP4，但流程仍然是说出意图、生成草稿、观看结果、继续反应和修正。也就是说，Agentic Engineering 不只适用于代码；只要产物能被拆成可描述、可生成、可审查的中间形态，就能进入同样的质量闭环。

所以，这一组技巧背后的共同点是：不要把 agent 的输出当成最终信号。近期社区讨论、会议原文、人的判断、视频成片、PR 里的 demo，才是更接近真实世界的信号。计划让任务开始，执行让任务流动，而原始信号负责把方向拉回来。缺少这一层，agent 越快，偏离也可能越快。

## 把一次经验沉淀成可复用系统

原文后半段开始谈 notes、Proof、skills、open source 和 Printing Press，主题从“这次怎么做成”转向“下次怎样少从零开始”。这也是 Compound Engineering 这个词里最容易被忽略的一半：不是每次都让 agent 更努力，而是让每次任务留下能复用的知识、接口和流程。

notes 是最朴素的一层复用。原文说，计划之所以越来越好，是因为 agent 能读到过去写过的计划；这个思路继续展开，就变成让 Bear、Obsidian、gbrain、supermemory 这类工具成为 agent 的知识库。重点不在某个笔记软件，而在选择有 CLI 或 API 的知识入口，让 agent 能读到过去的会议、想法、决策和半成品。这样下一次计划不只是基于当前提示词，而是带着长期上下文进入任务。

这和传统知识管理不完全一样。人类笔记常常服务于“以后再看”，但 agent 知识库服务于“以后能直接调用”。一条会议记录、一份旧 plan、一段半成形判断，如果只能躺在笔记软件里，就还是静态资料；如果 agent 能检索、引用、改写、纳入计划，它就变成了工作流的一部分。复用的关键，是让知识不只被保存，还能在下一次执行里被激活。

Proof 解决的是另一类复用：把 agent 产物带进人类协作。`plan.md` 对终端里的操作者很方便，但直接发给不在终端里工作的同事，阅读和评论都会变得别扭。原文把 Proof 放在这里，是因为它让计划能像文档一样被分享、评论，再把评论带回 agent 循环。也就是说，计划不再只是 agent 的内部任务单，而是能进入团队评审的协作界面。

skills 则是把重复动作固化成可调用能力。原文的规则很直接：一件事做超过两次，就考虑把它写成 skill。这里最重要的不是“会写一个 skill 文件”，而是把反复出现的判断、步骤、输入格式、验证方式从一次性提示词里拿出来，变成 agent 以后都能继承的工作方法。每一次 skill 化，都是把个人习惯从聊天记录里提取出来，变成系统能力。

开源贡献把复用继续往外推。原文提到，用同样的 `/ce-plan` + `/ce-work` 循环给 Python、Go、Vercel Agent Browser、OpenClaw 等项目贡献代码。表面看是个人产出变多了，深层价值却在于：当一个工作流能稳定给外部项目提交真实 PR，它就不再只是本地效率技巧，而开始接受维护者、社区、CI、review 的检验。开源项目会把“看起来能用”的方法逼成“别人也愿意合并”的质量。

Printing Press 则把复用从代码项目推到真实服务。原文把它描述成一组面向现实服务的 CLI，让 agent 可以处理预热车、Instacart 加购物品、监控比赛、查航班和积分这类日常任务。这里最值得保留的不是具体服务，而是“agent-native CLI”这个形态：把一个人经常手动操作的服务包装成 agent 能安全调用的接口，让自动化不再停留在网页理解或文本生成里。

所以，复用系统可以分成几层：notes 让长期知识可读，Proof 让计划可协作，skills 让重复方法可调用，open source 让工作流接受外部检验，Printing Press 让现实服务变成 agent 可操作的接口。一次任务如果没有留下任何东西，它只是完成了一次；如果留下计划、评论、skill、PR 或 CLI，它就在提高下一次的起点。

## 最后必须讲风险边界

原文越写到后面，越不像单纯的效率分享。权限、auth、硬件、电量、成本、心理状态、有没有用户，这些东西都开始浮出水面。它们提醒读者：Agentic Engineering 一旦真的跑起来，问题就不再只是“模型会不会写代码”，而是这套系统会消耗什么资源、触达哪些真实账户、影响哪些生活节奏，以及人是否还在控制方向。

第一个边界是权限。前面提到 YOLO 模式，是因为多会话执行如果总被确认框打断，会很难形成吞吐量。但权限放开以后，agent 能做的事情也会从“写一段草稿”变成“改文件、跑命令、调用服务、触发真实动作”。这要求工作流里必须有别的护栏：计划要清楚，git 要可回退，重要服务要有明确授权范围，高风险动作不能只靠自动化惯性推进。

第二个边界是 auth。Printing Press 和 Agent Cookie 最吸引人的地方，是让 CLI 使用真实浏览器会话，代表用户去操作现实服务。它也正因为这一点变得敏感：一旦 agent 拿到了登录态，能力就从“知道这个服务”变成“能以用户身份行动”。这类工具真正需要设计的，不只是调用是否方便，还有会话如何传递、哪些服务允许自动化、哪些动作必须人工确认、失败时怎样停止。

第三个边界是资源成本。原文提到新机器、六个 Claude 会话、Codex、远程机器、电池和充电器，不是炫耀装备，而是说明高吞吐 agent 工作流会真实吃掉硬件、电量、订阅费和注意力。多个 $200 级别的订阅、长时间运行的会话、远程机器和大量工具，并不会因为叫“自动化”就变成零成本。它们只是把成本从手动操作转移到了计算、账户、网络和维护上。

第四个边界更难讲：心理反馈。原文把 AI psychosis 单独列出来，是因为 agent 带来的即时反馈非常强。一个人突然觉得什么都能造，新的想法不断被 plan、被 build、被 demo，很容易进入持续亢奋的建造循环。真正危险的不是某个项目没有用户，而是人被这个循环吞掉，忽视身边的人，也不再认真问“这个东西有没有人需要”。

这并不是说只允许构建有市场的大产品。原文也承认，有些最好的东西本来就是自用工具。关键在于诚实：如果只是个人工具，就承认它是个人工具；如果想要用户，就必须离开创作亢奋，去面对真实发布、真实反馈和长期积累。Agent 让构建变快，但不会自动证明方向正确。更快地产出，并不等于更接近需求。

原文最后说，这篇文章本身就是用这套方式写出来的：语音输入、Claude Code、`cmux`、Proof、`last30days`，不断说出修改意图、生成、反应、再改。这个收尾很有意思，因为它让整篇文章不只是讲工作流，而是展示工作流。但也正因为如此，它应该被读成一个重度实践者的系统复盘，而不是无条件照抄的清单。

所以最后真正该带走的，不是“立刻打开所有权限、启动六个会话、把生活服务都接进 agent”。更稳的顺序是：先让任务有计划，再让执行可并行，接着用原始信号校准质量，把重复经验沉淀成系统，最后为权限、auth、成本和心理状态设边界。Agentic Engineering 的成熟标志，不是 agent 能跑多快，而是人在系统变快以后，仍然知道什么时候该踩油门，什么时候该刹车。

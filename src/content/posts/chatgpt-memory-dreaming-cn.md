参考来源：[OpenAI: Dreaming: Better memory for a more helpful ChatGPT](https://openai.com/index/chatgpt-memory-dreaming/)。

OpenAI 在 2026 年 6 月 4 日开始推出新的 ChatGPT 记忆系统。表面上看，这是一次个性化体验更新：ChatGPT 能更好地记住用户偏好、项目背景和限制条件。放到 Agent 工程里看，它更像一个信号：长期记忆正在从“用户手动保存几条事实”，变成一个会在后台整理、更新和过期的状态层。

## 从 Saved Memories 到 Dreaming

ChatGPT 的第一版记忆在 2024 年 4 月上线，核心是 `saved memories`。用户明确说“记住这件事”，系统才把它写进未来对话可用的记忆里。这种方式很容易理解，也便于用户管理，但问题也明显：没有被显式保存的上下文会丢掉，已经保存的事实又会随着时间变旧。

OpenAI 在 2025 年 4 月引入第一版 `dreaming`，让 ChatGPT 可以参考保存列表之外的历史对话，在后台自动整理记忆。2026 年这次更新，则把 dreaming 做成更强、更省计算的记忆架构。它不只是等用户下命令，而是从多次对话中合成一份更接近当前状态的记忆，让新对话不必总从零开始。

这个变化的关键，不是“记得更多”。更准确地说，是记忆从一组静态笔记变成了持续维护的状态。过去的系统像一个需要用户手动更新的备忘录；新的系统更像一个后台任务，定期判断哪些信息仍然相关，哪些信息应该被改写，哪些偏好应该继续影响下一次回答。

## 好记忆要同时处理三件事

OpenAI 这篇文章给了一个很实用的评估框架：好的记忆至少要能延续上下文、遵守偏好和限制、并随着时间保持新鲜。

第一件事是延续上下文。用户做长期项目时，不希望每次都重新介绍自己用什么设备、正在推进什么目标、之前做过哪些选择。对 Agent 来说，这相当于跨会话保留任务背景，否则每次执行都会退回“一次性问答”。

第二件事是遵守偏好和限制。比如用户是素食者，未来餐饮建议就不该继续推荐不合适的选项；用户要求某类话题少提，系统也应该把它当成约束。这类记忆不是装饰，而是后续行动的条件。

第三件事是时间。很多记忆有生命周期。“用户七月要去新加坡”和“用户已经从新加坡回来”不是同一条状态。一个不会过期的记忆系统，越努力记住，越容易在未来给出错误建议。

## 对 Agent 设计的启发

Agent 需要的记忆不该只是更大的上下文窗口。上下文窗口解决的是“这次对话能塞多少东西”，记忆系统解决的是“跨任务、跨时间、跨工具后，哪些状态应该留下来”。这两件事很容易混在一起，但工程边界不同。

代码 Agent 也有类似问题。`AGENTS.md` 适合保存稳定规则，比如测试命令、发布流程和仓库约定；自动化记忆适合记录上一次筛选过哪些来源；一次任务里的临时报错则不一定值得长期保存。如果三类信息都被塞进同一个长期记忆里，下一次执行就可能被旧分支、旧报错或旧假设带偏。

所以，真正值得关注的不是 Dreaming 这个名字，而是它背后的产品判断：长期记忆需要合成、更新、审计和用户修正。OpenAI 也提供了 memory summary 页面，让用户查看和调整系统认为自己知道的内容。这一步很重要，因为只要记忆开始自动合成，问题就不再只是“能不能记住”，还包括“用户能不能看见它怎么理解自己”。

## 更大的问题是可控性

Agent 越像长期协作者，记忆越会变成能力的一部分。它能减少重复解释，也能让回答更贴近真实约束。但记忆一旦出错，影响也会持续存在：错误偏好会被反复套用，过期事实会继续参与决策，模糊总结会悄悄改变系统对用户的理解。

因此，记忆系统至少需要四个边界：来源可追，知道某条记忆来自哪里；作用域清楚，区分个人偏好、项目状态和一次性任务信息；时间可更新，让过期事实自然降权或改写；用户可纠正，让系统的长期画像不是黑箱。

这篇 OpenAI 更新最值得带走的判断是：Agent 记忆不是“多存一点聊天记录”，而是一个会影响未来行动的状态管理系统。它越强，就越需要像数据库、权限和评测一样被认真设计。

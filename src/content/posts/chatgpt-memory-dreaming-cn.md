OpenAI 在 2026 年 6 月 4 日开始推出新的 ChatGPT 记忆系统。原文表面上是在讲 ChatGPT 如何更好地个性化回答：记住用户偏好、项目背景、旅行计划、写作习惯和限制条件。放到 Agent 工程里看，这件事更像一个架构信号：长期记忆正在从“用户手动保存几条事实”，变成一个会在后台合成、更新、过期和接受审计的状态层。

读这篇原文时，关键不在于记住又一个产品功能名，而在于看清 Agent 记忆的性质变化：一旦记忆会影响未来回答和行动，它就不再只是聊天体验优化，而是类似数据库、权限和评测一样的基础设施问题。

## 第一代记忆：手动保存的事实列表

ChatGPT 的第一版记忆在 2024 年 4 月上线，核心是 `saved memories`。用户可以明确要求系统记住某件事，比如职业背景、长期偏好、正在进行的项目，之后这些信息会进入未来对话。这个设计的优点很明显：用户容易理解，也能在设置里查看和管理。

  <figcaption>`saved memories` 更像一组被保存下来的事实列表，便于查看，但也容易变成静态备忘录。</figcaption>
</figure>

问题也在这里。手动保存的事实列表很适合记录稳定信息，却很难承载长期上下文的变化。用户不一定每次都会说“请记住这个”；即使记住了，事实也可能随着时间变旧。一个旅行计划、一个短期项目、一个阶段性偏好，过一段时间后可能不该继续影响回答。

这也是传统“记忆”最容易被误解的地方。它看起来像在增加上下文，实际更像在维护状态。如果状态只会增加、不会被更新或降权，系统越努力记住，越可能在未来带着过期前提继续回答。

## Dreaming：从聊天历史里合成当前状态

OpenAI 在 2025 年 4 月引入第一版 `dreaming`，让 ChatGPT 不只依赖显式保存的记忆，也能参考历史对话，在后台整理出仍然有用的长期信息。2026 年这次更新，则把 dreaming 做成更强、更省计算的新架构，并逐步扩展到更多用户。

这个变化的重点不是“记得更多”，而是“如何合成”。`saved memories` 像一份用户和系统共同维护的笔记；`dreaming` 更像一个后台整理任务：它从多次对话里判断哪些信息仍然代表当前用户，哪些信息只是一次性任务，哪些旧信息需要改写或过期。

原文提到，改进后的 dreaming 架构把服务记忆所需的计算量降低到原来的约五分之一。这个数字不只是性能优化，也说明记忆系统已经从实验性体验进入平台化阶段：只有成本下降，长期记忆才可能从 Plus / Pro 用户扩展到更广泛的用户群。

## 好记忆至少要解决三个问题

OpenAI 原文把记忆质量拆成几个方向，其中最值得保留的是三件事：延续上下文、遵守偏好和限制、随时间保持新鲜。

第一是延续上下文。用户做长期项目时，不希望每次都重新介绍背景。比如一个人反复讨论摄影设备、旅行目的地、文章风格或代码仓库，下一次提问时，模型如果能带着这些背景进入任务，回答就会少很多模板化内容。对 Agent 来说，这相当于跨会话保留任务状态，否则每次执行都会退回一次性问答。

第二是遵守偏好和限制。OpenAI 用新加坡旅行计划举例：如果系统知道用户喜欢野生动物摄影、需要空调较强的酒店、偏好安静场景，那么推荐就不该只是普通景点清单，而应该把这些偏好变成规划约束。这里的记忆不是装饰性个性化，而是在改变后续任务的搜索空间。

第三是时间。很多记忆都有生命周期。“用户七月要去新加坡”和“用户已经从新加坡回来”不是同一条状态；“用户正在评估某个框架”和“用户已经选定并上线”也不是同一条状态。如果系统不能识别阶段变化，记忆就会变成旧前提的放大器。

这三个问题合在一起，说明记忆系统不是简单缓存。缓存关心命中率，记忆还要关心适用范围、时间衰减和冲突更新。

这一步很关键。手动保存的记忆，用户大致知道它从哪里来；自动合成的记忆，则更容易变成一个看不见的长期画像。如果用户无法查看、纠正、关闭或删除，记忆就会变成黑箱偏置：它可能让回答更贴心，也可能让错误偏好被持续套用。

对 Agent 产品来说，用户控制不该被放在最后。记忆越强，越要先设计可见性和纠错机制。否则系统一旦误解用户，后续每一次回答都会在同一个误解上继续优化。

## 对 Agent 工程的启发

Agent 需要的记忆，不应该只理解成更大的上下文窗口。上下文窗口解决的是“当前任务能塞多少信息”，记忆系统解决的是“哪些状态应该跨任务留下来”。前者是容量问题，后者是生命周期问题。

放到代码 Agent 或工作流 Agent 里，这个区别更明显。`AGENTS.md` 适合保存稳定规则，比如测试命令、发布流程和仓库约定；自动化记忆适合记录上一次筛选过哪些来源；一次任务里的临时报错、临时分支状态和失败尝试，不一定值得长期保存。如果三类信息都被塞进同一个长期记忆里，下一次执行就可能被旧分支、旧报错或旧假设带偏。

所以，一个可用的 Agent memory 至少要有四条边界：

- 来源可追：知道这条记忆来自哪次对话、哪个项目或哪类用户行为。
- 作用域清楚：区分个人偏好、项目状态、团队规则和一次性任务信息。
- 时间可更新：让过期事实降权、被改写，或在新证据出现时失效。
- 用户可纠正：允许用户查看系统摘要，明确删除或修正错误理解。

这也能和前面几篇 Agent 文章接起来。ReAct 解决的是任务内的观察回路：模型在行动后如何根据环境反馈调整下一步。Agent eval 解决的是结果验证：系统是否真的把世界状态改对了。Memory 则解决跨任务状态：哪些经验、偏好和约束应该进入下一次任务，哪些又应该被遗忘。

如果只保留上下文，不检查结果，Agent 会重复错误；如果只看结果，不维护长期状态，Agent 会每次从零开始；如果记忆无法审计，Agent 会把旧假设包装成“懂你”。这三者必须一起设计。

## 越会记，越需要边界

OpenAI 这次更新最值得带走的判断，不是 ChatGPT 终于“更有记性”，而是长期记忆正在成为 AI 产品的基础状态层。它能减少重复解释，也能让回答更贴近真实约束；但它一旦出错，影响也会持续存在。

这对 Agent 产品尤其重要。一个长期协作者如果记错了用户偏好、项目阶段或权限边界，问题不是一次回答不准，而是后续一整串行动都会建立在错误状态上。记忆越强，越需要像数据库一样考虑来源、作用域、更新和删除；越像协作者，越需要像权限系统一样暴露控制入口。

因此，Agent 记忆不应被当成“多存一点聊天记录”。更合理的理解是：它是一套会影响未来推理和行动的状态管理系统。系统越有能力在后台整理用户，就越应该让用户看见它整理出了什么。

一次复杂重构最容易失控的地方，往往不是第一行代码怎么改，而是同一时间有太多事情需要被记住。哪些入口还没迁完，哪些测试只是偶然通过，哪些文件是旧实现的影子，哪些改动看起来正确但其实绕过了业务约束。让一个人从头做到尾当然可以，但任务越长，这个人越容易陷在自己的上下文里：先相信自己刚刚做过的判断，再沿着这个判断继续往下补。

工程团队处理这类任务时，很少只靠一个人闷头干完。更常见的做法是临时搭一个工作台：有人先盘点边界，有人负责改实现，有人专门挑错，有人只看测试和回归风险。等任务结束，这个结构可以拆掉，因为它不是组织架构的一部分，只是为这次任务搭出来的临时 harness。

Claude Code 的 dynamic workflows 讲的正是 Agent 版本的这件事。普通 coding task 交给默认会话就够了：模型读上下文、改文件、跑命令、根据结果继续调整。但复杂任务不只是“多做几步”。它需要把计划、执行、验证和汇总拆成不同角色，让每个角色拿到更窄的上下文和更明确的职责。dynamic workflow 不是给一个 prompt 加长尾巴，而是为当前任务临时生成一个能运行的执行壳。

## harness 补的是任务结构

这里的 `harness` 可以先理解成“让模型开始工作的运行框架”。它决定模型看到什么上下文，能调用哪些工具，怎样把中间结果传回主流程，失败时是否能重试，结束后怎么判断任务有没有真的完成。单看模型能力，很多 Agent 都像是在“会写代码”；但一旦把任务放进真实仓库，harness 的设计会直接决定它是稳稳推进，还是在越来越长的上下文里打转。

默认 harness 适合大多数普通任务。比如修一个明确 bug、补一个小测试、改一段文案，模型可以在同一个上下文里读文件、编辑、运行检查，再根据报错修正。这个路径启动成本低，也避免了为了简单任务引入额外编排。

固定 workflow 是另一种形态。团队提前写好流程，让 Agent 按既定步骤执行。它适合稳定、重复、边界清楚的任务，比如固定代码审查、安全检查、发布前核对。问题在于流程写死以后，任务就必须去适应流程。遇到形状不同的问题，要么流程显得太重，要么关键分支没有被覆盖。

dynamic workflow 把生成流程这件事推到任务现场。Claude Code 可以根据当前任务写一段临时 workflow，用代码组织 subagents、模型选择、隔离环境、结果汇总和停止条件。这里重要的不是“workflow 用 JavaScript 写”这个表面细节，而是 workflow 成了任务的一部分：需要并行研究，就生成并行研究结构；需要挑错，就生成 verifier；需要比较多个候选方案，就生成 tournament 或 generate-and-filter。

| 形态 | 适合的任务 | 优点 | 主要风险 |
| --- | --- | --- | --- |
| default harness | 明确、短链路、反馈快的普通 coding task。 | 启动成本低，读、改、测可以在同一上下文里连续推进。 | 任务变长后容易被早期判断和上下文噪声带偏。 |
| static workflow | 稳定、重复、边界清楚的流程。 | 流程可复用，结果更容易比较。 | 非典型任务要么被过度处理，要么漏掉关键分支。 |
| dynamic workflow | 长任务、宽任务、对抗性任务，或者需要比较多个候选方案的任务。 | 可以为当前任务临时拆角色、隔离上下文、汇总结果。 | 更消耗 token 和运行时间，也更依赖清楚的预算与停止条件。 |

这个区别看起来细，但会改变使用 Agent 的姿势。复杂任务里，问题不止是模型能不能答对，而是谁来组织任务链：谁先看问题，谁展开，谁反驳，谁汇总，失败后回到哪里。没有这层结构，复杂度会被塞回同一个聊天窗口，最后仍然由人脑兜底。

## dynamic workflow 真的怎么跑

dynamic workflow 不是让用户手写一套编排框架。更自然的入口，是直接告诉 Claude Code 任务需要怎样的工作方式：比如需要调查一个 flaky test，先从最近测试记录里找模式，再分派 subagent 看代码、日志和历史改动，最后由另一个角色挑战结论。Claude Code 根据这个任务写 workflow，并在运行时负责启动 subagents、收集中间结果、处理恢复和汇总。

这也是为什么它和普通“多开几个 agent”不一样。workflow 里可以明确每个角色的输入、输出和预算：研究角色只读资料，验证角色只检查证据，综合角色只把结构化结果收回来。角色之间不靠聊天窗口里一大段松散上下文传话，而是靠 workflow 里的数据结构交接。

| 组成 | 在 workflow 里的作用 | 如果缺失会怎样 |
| --- | --- | --- |
| 子任务拆分 | 把大任务分给不同 subagents。 | 单个上下文越来越长，早期判断会反复影响后续动作。 |
| 模型选择 | 给不同阶段选择合适模型和推理强度。 | 所有步骤都用同一种成本和能力，简单步骤浪费，关键步骤不够稳。 |
| 隔离环境 | 让子任务在独立上下文或 worktree 里运行。 | 并行结果互相污染，文件改动和测试状态容易踩在一起。 |
| 结构化输出 | 要求每个角色按同一格式返回证据、结论和不确定点。 | 汇总阶段只能拼自然语言，最后像会议纪要而不是判断。 |
| 停止条件 | 定义成功、失败、预算耗尽和需要人接管的边界。 | `loop until done` 很容易跑成 `loop until tired`。 |

一个写得好的 workflow，不会让每个 subagent 都“自由发挥”。它更像一个小型任务程序：输入是什么，谁处理哪一段，输出按什么格式回来，哪个环节允许重试，哪个环节必须停止。模型仍然在工作，但工作不再只发生在一条长回复里。

## 几种常见模式

这些模式看起来像 prompt 技巧，但更适合被理解成任务结构。差别不在于多叫了几个 subagent，而在于每种结构回答了一个不同问题：任务该先分类，还是先并行展开；结果该由谁合成；哪些结论必须被反向验证；什么时候应该继续循环，什么时候应该停。

| 模式 | 适合的问题 | 核心动作 | 最容易出错的地方 |
| --- | --- | --- | --- |
| `classify-and-act` | 输入很多、类型不同、处理路径不同。 | 先分类，再按类别选择行动。 | 分类标准不清楚时，后面的行动会从第一步就偏掉。 |
| `fan-out-and-synthesize` | 大任务可以拆成许多相对独立的小块。 | 多个 agent 分头处理，再由汇总阶段合成结果。 | 汇总阶段没有结构化输入时，只会把分散结论拼成流水账。 |
| `adversarial verification` | 结论需要被怀疑、审计或按 rubric 检查。 | 给每个结果配一个反向验证角色。 | verifier 如果只复述标准，不检查证据，就会变成形式化盖章。 |
| `generate-and-filter` | 需要大量候选方案，但最终只要少数高质量结果。 | 先生成，再按标准过滤、去重、验证。 | 生成太松会制造噪声，过滤太窄会错过有潜力的方案。 |
| `tournament` | 候选对象需要按质量、适配度或偏好排序。 | 用两两比较或分组比较逐步筛出优胜者。 | rubric 太主观时，比较过程会放大评审者偏见。 |
| `loop until done` | 任务有明确停止条件，但不知道要尝试多少轮。 | 反复执行、检查、修正，直到满足条件或预算耗尽。 | 没有预算和退出条件时，循环会把 token 和时间吃空。 |

`fan-out-and-synthesize` 是最容易用起来的一种。把一个大任务拆成多个小任务，每个 agent 拿到干净上下文，只负责自己的部分，最后统一合成。它适合迁移、代码库调研、批量文件检查、资料收集。关键不在“并行”本身，而在合成阶段：每个 agent 必须返回同一种结构，汇总才有可能判断差异，而不是复述大家各说了什么。

`adversarial verification` 解决另一类老问题：Agent 很容易相信自己刚写出来的东西。一个实现 agent 刚修完 bug，就倾向于解释为什么它已经修好了；一个研究 agent 刚整理完资料，也容易把薄弱来源写成确定结论。把 verifier 单独拆出来，是让它站在相反方向看证据：这个 patch 有没有漏边界，这条事实有没有来源，这个方案有没有违反约束。

`generate-and-filter` 和 `tournament` 更适合创意、命名、设计方案、候选排序。它们承认第一轮产物会很粗糙，所以先扩大候选池，再用 rubric 或两两比较把结果收紧。很多偏好型任务本来就需要比较才能看出差异，要求模型一次给出“最佳答案”，反而会把判断过程藏掉。

`loop until done` 把 workflow 变成带停止条件的循环。比如复现一个偶发测试、持续 triage 一批 issue、反复验证一组规则，Agent 不应该跑一轮就宣布结束，而应该在每轮之后检查证据是否足够、是否需要换假设、是否已经到达预算边界。这个模式最能暴露 workflow 的工程属性：没有预算、状态记录和退出条件，循环就不是自动化，而是失控。

## 它适合哪些任务

dynamic workflow 最适合的任务，通常有一个共同点：单个上下文窗口很难同时装下全部状态、证据和反对意见。迁移和重构是典型例子。一个模型可以读一处调用点并改掉它，但大规模 rename、跨模块迁移、旧接口清理，往往需要先分片盘点，再并行修改，随后由验证角色检查有没有漏掉旧路径。这里的 worktree 隔离很重要，因为多个 agent 同时改仓库，如果没有隔离，就会互相踩文件和测试状态。

深度研究也很适合。研究任务不是简单搜索几篇文章，而是要拆成检索、追源、交叉验证和综合判断。一个 agent 专门找资料，另一个 agent 检查来源质量，再由最后一个角色汇总成报告，这比单个模型一边搜一边写更容易保持证据链。Claude Code 的 deep research skill 走的就是这个方向：并行搜索、抓取来源、验证 claims，再合成带引用的报告。

还有一些任务乍看不像 coding，也很需要这层结构：批量排序、triage、根因调查、记忆和规则执行检查、跨模型评估、商业计划评审、CLI 命名。它们共同的问题不是“答案很长”，而是评价标准、证据和候选对象很多。如果让一个模型在同一段上下文里一次性完成，越到后面越容易凭感觉补齐。

| 任务 | workflow 可以怎样拆 | 需要盯住的边界 |
| --- | --- | --- |
| 大规模迁移或重构 | 盘点调用点、分片修改、验证旧路径、汇总剩余风险。 | worktree 隔离和最终合并策略。 |
| 深度研究 | 检索、追源、事实核查、综合报告。 | 引用质量和 claim 级验证。 |
| 深度验证 | 抽取草稿或 patch 里的 claims，逐条查证。 | verifier 必须看证据，不只看语言是否顺。 |
| 排序和筛选 | 先分类、去重，再按 rubric 或 tournament 比较。 | rubric 要先写清楚，否则比较只是偏好投票。 |
| 根因调查 | 分别检查日志、代码、数据、历史变更，再挑战假设。 | 不要让第一个看起来合理的解释过早收敛。 |
| 规则和记忆执行 | 检查 agent 是否遵守项目规则、历史教训和用户偏好。 | 规则需要可定位，不能只写成抽象价值观。 |
| 主观评审 | 拆成用户视角、产品视角、工程视角或投资人视角。 | 不要把多视角包装成客观真理。 |

这里最有意思的一点，是 workflow 让“判断”变成显式对象。过去可能只是在 prompt 最后写一句“请认真检查”；现在可以给检查单独分配角色、输入、输出和预算。这个变化看似很小，落到复杂任务里，往往就是能不能把错误拦在下一步之前的差别。

## 别把它用成重型仪式

dynamic workflow 的反面，不是“简单任务不重要”，而是很多任务根本不需要额外编排。一个小 bug、一个明确测试失败、一次局部文案修改，默认 harness 已经能快速完成。如果这类任务也套上五个 reviewer、多个 worktree 和长循环，系统看起来更认真，实际只是把反馈链路拉长。

| 任务信号 | 更适合 default harness | 更适合 dynamic workflow |
| --- | --- | --- |
| 任务长度 | 一两个文件、一次清楚修改、马上能验证。 | 很多文件、很多轮尝试，或者需要持续恢复上下文。 |
| 子问题关系 | 子问题强耦合，拆开后反而增加沟通成本。 | 子问题可以分片处理，最后能结构化合成。 |
| 验证方式 | 一个测试、一次构建或一次人工阅读就能判断。 | 需要多个角度互相验证，或者要逐条核对证据。 |
| 成本约束 | token、时间、机器资源都应该尽量省。 | 任务价值足够高，值得花更多 compute 换稳定性。 |
| 失败后果 | 错了也容易回滚或重写。 | 错误会扩散到大量文件、结论、候选对象或真实操作。 |

成本是第一个边界。dynamic workflows 往往会使用更多 token，也会带来等待时间、机器资源和人的注意力成本。复杂工作流如果没有预算，可能在研究、验证、重试之间越跑越远；如果没有清楚的停止条件，`loop until done` 会变成 `loop until someone gets tired`。workflow prompt 里应该明确预算、成功标准和退出条件，而不是只写“认真检查”。

权限和隔离也不能省略。dynamic workflow 经常会启动多个 subagents，甚至让它们在各自 worktree 里修改文件。这样能减少上下文污染，但也引入合并、资源争用和权限扩散的问题。尤其是 triage 或外部输入场景，读取不可信内容的 agent 不应该直接拥有高权限动作。把“读”和“行动”拆开，本身就是 workflow 设计的一部分。

还有一种情况要先停住：任务只是看起来复杂，其实核心问题没有被定义。比如“帮这个产品变好”“把这篇文章写得更高级”“把仓库整理一下”。这类请求如果直接交给 dynamic workflow，很容易并行产出一堆看似有结构的建议，但成功标准仍然悬空。更合适的顺序是先用计划或讨论把问题定清楚，再决定是否需要 workflow。

## 保存成 skill 之前，先让它跑过一次

Claude Code 支持把有用的 workflow 保存起来，再在后续任务里复用，甚至和 team skill 一起分发。这很自然：如果某个 research workflow、verification workflow 或 migration workflow 已经跑出了稳定效果，就不该每次从零描述。workflow 被沉淀后，可以变成团队共享的任务 harness。

但保存之前最好先让它在真实任务里跑过一次。很多 workflow 在纸面上很漂亮，真正运行时会暴露一些朴素问题：subagent 输出格式不一致，汇总阶段不知道该信谁，预算不够，停止条件太模糊，或者某个角色拿到了不该拿的权限。这些问题不跑一遍，很难只靠设计发现。

这里可以把 skill 和 workflow 分清：skill 更像可复用能力包，里面写触发条件、项目约定、脚本和参考材料；dynamic workflow 更像任务现场生成的编排层。一个成熟的 Agent 工作流里，两者应该互补。skill 提供稳定知识，workflow 决定这次怎样组合这些知识、哪些步骤要并行、哪些结论要被验证。

也不要把 workflow 保存当成收尾动作的仪式感。只有当它在多个相近任务里都有复用价值，保存才有意义。否则把一次性脚手架塞进长期 skill，只会让后面的 agent 多读一层过期结构。

## 对 Agent 工程的影响

把 dynamic workflow 放回 Agent 工程里看，它补的是 ReAct 没有直接覆盖的那一层。ReAct 关心的是一次任务内部如何交替思考、行动和观察：模型什么时候调用工具，工具返回后怎样更新判断。dynamic workflow 关心的是更高一层的问题：当一个任务需要多个角色、多段上下文、多条候选路径时，谁来组织这些行动，谁来保存中间状态，谁来决定下一轮该不该继续。

它和 eval 也接得上。Agent eval 里常说要看 outcome，而不是只看 transcript。dynamic workflow 不能替代 eval，但它能让 outcome 更容易被制造和检查。一个 verifier agent 可以逐条检查事实，一个 comparison agent 可以按 rubric 比较候选，一个 triage agent 可以把结果升级给人。workflow 把“要验证”变成一个明确角色，而不是最后附上一句“请检查一下”。

从产品设计角度看，dynamic workflow 说明 Agent 的下一层竞争不只是模型会不会更强，而是运行时能不能为任务生成合适的组织形态。普通任务需要低摩擦；复杂任务需要隔离、并行、验证、预算和恢复；高风险任务还需要权限分层和可追溯日志。如果所有任务都塞进同一个聊天窗口，复杂度迟早会从用户脑子里漏出来。

复杂任务里，多几个模型一起做没有太大意义。能减轻压力的，是让每个角色在任务结构里有明确位置：有人展开问题，有人专门挑错，有人负责合成，也有人判断该不该继续。dynamic workflow 的价值就在这里。它把一次 prompt 里的美好设想，挪进了一条真的能运行、能停下、能复查的任务链。

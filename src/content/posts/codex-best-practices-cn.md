参考来源：[OpenAI Developers: Best practices - Codex](https://developers.openai.com/codex/learn/best-practices)。

这篇不是逐字翻译，而是按中文博客的阅读习惯重写的一版导读。原文最重要的观点其实很简单：不要把 Codex 当成一次性问答工具，而要把它当成一个可以被配置、训练和复盘的工程队友。

## 先把任务讲清楚

Codex 可以在提示词不完美的时候也做出有用结果，但在复杂仓库里，清晰上下文会显著减少误解。一个好请求通常包含四块信息：

- 目标：这次到底要改什么、修什么、做什么。
- 上下文：相关文件、目录、报错、设计文档、历史实现在哪里。
- 约束：架构边界、编码规范、安全限制、不能碰的地方。
- 完成标准：什么时候算完成，比如测试通过、行为改变、bug 不能复现。

这四项不是形式主义。它们的作用是帮 Codex 收窄搜索范围，让它知道哪些假设可以做、哪些地方必须先验证。

推理强度也应该跟任务难度匹配。小改动、明确 bug、单文件修复可以偏低；跨模块重构、疑难调试、长链路任务则应该提高推理强度。不要把所有任务都开到最高，也不要在需要系统性分析时只追求快。

## 难题先规划

遇到复杂、模糊、多步骤的任务，先让 Codex 规划，再让它实现。规划阶段的价值不只是列步骤，而是让它先读仓库、补齐上下文、发现不确定点。

几种实用做法：

- 用 `/plan` 或计划模式，让 Codex 先探索再动手。
- 如果想法还很粗，让 Codex 先反问你，把模糊需求问成可执行任务。
- 长任务可以用 `PLANS.md` 这类模板，把目标、阶段、验证方式固定下来。

一个经验是：如果你自己都说不清验收标准，就先不要急着让 agent 写代码。

## 把重复规则放进 AGENTS.md

如果某些提示词每次都要重复，它们就不应该继续留在聊天框里。`AGENTS.md` 更适合承载稳定规则，比如：

- 仓库结构和关键目录。
- 本地启动、构建、测试、lint 命令。
- 代码风格、PR 习惯、审查要求。
- 安全边界和明确的 do-not 规则。
- 对“完成”的定义，以及如何验证。

`AGENTS.md` 可以分层：个人默认规则放在全局位置，项目共享规则放在仓库根目录，更细的目录可以有局部规则。越靠近当前工作目录的规则优先级越高。

它不需要很长。短、准、能落地的规则，比一大段抽象价值观更有效。一个好维护方式是：当 Codex 第二次犯同类错误时，让它做一次复盘，再把复盘里真正稳定的规则沉淀进去。

## 配置要贴近真实工作流

很多“agent 不靠谱”的问题，本质上是环境没配好：工作目录不对、没有写权限、默认模型不合适、缺少必要工具、外部系统连不上。

Codex 的配置可以分成三类：

- 个人默认配置：放在 `~/.codex/config.toml`。
- 仓库级配置：放在 `.codex/config.toml`。
- 一次性覆盖：CLI 场景下用命令行参数处理临时需求。

配置里可以管理模型、推理强度、沙箱、审批策略、profile、MCP 服务等。新手最好先用较保守的权限，等你理解工作流后，再针对可信仓库或固定任务放宽。

## 让 Codex 自己验证结果

不要只要求 Codex “改完”。更好的要求是：改完后补测试、跑相关检查、确认行为、再审一遍 diff。

你可以把验证要求写进单次提示词，也可以沉淀进 `AGENTS.md`。常见检查包括：

- 为行为变化新增或更新测试。
- 运行对应测试套件。
- 跑类型检查、lint、格式化检查。
- 对照需求确认最终行为。
- 审查 diff 里是否有回归、风险模式或无关改动。

`/review` 也很适合做收尾，可以审未提交改动、某个 commit，或者按自定义审查清单检查。如果团队有 `code_review.md`，可以从 `AGENTS.md` 引用，让审查标准在不同仓库里保持一致。

## 外部上下文交给 MCP

当 Codex 需要的信息不在仓库里，就不要反复手动复制粘贴。MCP 的价值在于把外部系统变成工具连接进来，例如 issue、文档、浏览器、数据库、监控、内部系统。

适合使用 MCP 的场景：

- 上下文在仓库之外。
- 数据经常变化。
- 希望 Codex 调工具，而不是只读你粘贴的一段文字。
- 这个集成会在多个任务、多人或多个项目中重复使用。

不过工具不是越多越好。先接一两个真正能消除重复手工流程的工具，等流程稳定后再扩展。

## 把重复工作做成 Skill

如果一个流程已经能稳定复用，就不该继续依赖长提示词。Skill 适合把一类任务的方法、上下文、输入输出、辅助脚本打包起来。

一个好 Skill 通常满足这些条件：

- 只解决一个清晰问题。
- 有两三个具体用例。
- 输入和输出明确。
- 描述里写清楚什么时候该用它。
- 只在能提升可靠性时加入脚本或素材。

适合做成 Skill 的任务包括日志排查、发布说明草稿、按清单做 PR review、迁移规划、遥测摘要、标准调试流程等。先从本地个人 Skill 开始迭代，稳定后再考虑做成团队共享插件。

## 稳定流程再自动化

Automations 适合已经稳定的重复任务，比如定期总结 commit、扫描可疑 bug、生成 release notes、检查 CI 失败、整理 standup 摘要。

这里有个顺序：Skill 定义方法，Automation 定义时间表。一个流程如果还需要你频繁纠偏，就先做成 Skill 并打磨；等它足够可预测，再交给自动化。

自动化也不只适合执行任务，还适合维护工作流本身。比如定期回看最近的 Codex 会话，总结反复出现的问题，再改进提示词、`AGENTS.md` 或项目配置。

## 长任务要管好线程

Codex 的会话不是普通聊天记录，而是会积累上下文、决策和操作历史的工作线程。线程管理不好，时间一长就会变成噪声。

比较稳的习惯是：一个线程服务一个连贯任务。如果问题还属于同一条工作线，留在原线程能保留推理路径；如果方向真的分叉，再 fork 出新线程。

CLI 里一些命令也很实用：

- `/resume`：恢复历史会话。
- `/fork`：从当前会话分叉。
- `/compact`：压缩长上下文。
- `/agent`：在并行 agent 之间切换。
- `/status`：查看当前会话状态。

对于边界清楚的探索、测试、排查任务，也可以交给 subagent，让主线程继续聚焦核心问题。

## 常见误区

刚开始用 Codex 时，最容易踩的坑大概是这些：

- 把长期规则塞进单次提示词，而不是放到 `AGENTS.md` 或 Skill。
- 不告诉 Codex 怎么构建、测试、验证，导致它看不到自己的结果。
- 多步骤任务跳过规划，直接让它改代码。
- 在不了解工作流前给过高权限。
- 多个线程同时改同一批文件，却不用 git worktree 隔离。
- 手工流程还不稳定，就急着做自动化。
- 一步步盯着 agent 做事，而不是把它当成可以并行推进的工程队友。
- 一个项目只开一个大线程，最后上下文膨胀，质量越来越差。

归根到底，Codex 的最佳实践不是“写一个神级提示词”。更像是搭一套工程协作系统：一次任务讲清楚，稳定规则沉淀下来，外部上下文工具化，重复流程技能化，成熟工作自动化。

参考来源：[ReAct: Synergizing Reasoning and Acting in Language Models](https://arxiv.org/abs/2210.03629)，以及 [Google Research 对 ReAct 的介绍](https://research.google/blog/react-synergizing-reasoning-and-acting-in-language-models/)。

## 今天的 Agent 问题，ReAct 已经提前点出来了

今天做 Agent 系统时，讨论很容易先落到工具清单上：搜索、浏览器、代码执行、文件系统、数据库、工作流编排。工具越多，系统看起来越像能做事。但真正难的不是让模型“拥有工具”，而是让模型在任务推进过程中知道什么时候该想、什么时候该查、什么时候该行动，以及行动之后如何根据环境反馈修正下一步。

ReAct 论文切中的正是这个问题。它没有把语言模型只当成一个回答问题的文本生成器，也没有把工具调用处理成外部插件式的补丁，而是把推理和行动放进同一条轨迹里：模型先形成中间判断，再选择动作，拿到观察结果后继续更新判断。这个循环后来被很多 Agent 系统继承下来，只是表现形式不再局限于 prompt 里的 `Thought`、`Action`、`Observation` 字段，而是落到了工具调用协议、运行日志、浏览器状态、代码补丁和评测 transcript 这些工程对象里。

论文里的轨迹本身很朴素。以 HotpotQA 和 FEVER 这类需要查 Wikipedia 的任务为例，作者给模型的动作空间主要是搜索实体、在页面里查找字符串、最后提交答案。轨迹大致长成这样：

```text
Question: ...
Thought 1: 先判断需要查哪个实体。
Action 1: Search[...]
Observation 1: 工具返回相关页面片段。
Thought 2: 根据返回内容决定下一步查什么。
Action 2: Lookup[...]
Observation 2: 工具返回命中的句子。
Thought 3: 信息足够，可以合成答案。
Action 3: Finish[...]
```

这里的 `Thought` 是模型写给自己的中间推理，`Action` 是模型选择的外部动作，`Observation` 则不是模型生成的，而是环境或工具返回的结果。今天的 function calling、浏览器 Agent、代码 Agent 和 eval transcript，看起来已经比这个格式复杂很多，但底层仍然在处理同一个问题：模型什么时候该继续想，什么时候该对外部世界做一次动作，以及动作返回之后如何更新下一步判断。

从今天回看，ReAct 的重要性不在于它给 prompt 起了几个固定字段名，而在于它把 Agent 的核心矛盾讲清楚了：语言模型的内部推理如果不能接触外部世界，很容易停在自洽但不可靠的解释里；外部行动如果没有推理轨迹支撑，又会变成一串难以归因的工具调用。Agent 要稳定工作，需要把这两者接起来，并让中间过程足够可观察。

接下来先从几个 Agent 工程问题进入 ReAct：为什么只让模型思考还不够，工具调用如何变成推理链的一部分，轨迹又怎样帮助调试和失败恢复。等这些问题讲清楚之后，再回到这套范式没有直接解决的成本和边界。

## 只让模型思考，为什么不够

ReAct 不是凭空出现的。它接在 Chain-of-Thought 这条线后面：先让模型写出中间推理步骤，再给出最终答案。CoT 的价值很明显，尤其是在数学题、常识推理、多跳问答这类任务里，它能把原本藏在最终答案背后的推理过程摊开，让模型有机会分解问题、保存中间结论，也让人更容易看出答案是怎么来的。

但 CoT 仍然主要发生在模型内部。prompt 里给了什么信息，模型就围绕这些信息继续推演；模型记忆里有什么知识，它就从这些知识里组织解释。问题在于，很多 Agent 任务的关键事实并不在 prompt 里，也不能只靠模型参数可靠还原。网页现在是什么状态，数据库里有没有这条记录，某个 API 返回了什么，代码改完测试是否通过，这些都需要对外部环境做一次观察。

只让模型思考，最容易出现的失败不是“不会写推理过程”，而是推理过程建立在错误或过期的前提上。模型可以把一个不存在的事实写得很连贯，也可以在第一步误判之后继续把后面的步骤补完整。对读者来说，这条链路看起来有结构；对真实任务来说，它没有被环境校验过。推理越长，未必越可靠，有时只是把错误包装得更完整。

ReAct 论文把这个边界说得很清楚：reasoning traces 可以帮助模型制定、跟踪和更新行动计划，但 actions 才能让模型接触外部来源，拿到额外信息。换成今天的 Agent 工程语言，就是模型不能只在上下文窗口里闭环。它需要在关键节点停下来，把某个判断交给搜索、工具、代码执行或环境状态验证，然后再根据返回结果继续推进。

所以，ReAct 不是否定 CoT，而是补上 CoT 缺少的观察回路。单纯的 CoT 更像是在纸上推演；ReAct 则要求模型在关键节点把推演接到外部世界，看看前提是否成立、信息是否足够、下一步是否该换方向。这也是为什么它后来会自然连接到工具调用和 Agent runtime：Agent 真正需要的不是更长的独白，而是能被外部反馈打断和修正的推理过程。

## 工具调用不是外挂，而是推理链的一部分

如果把工具调用理解成“模型答不出来时临时查一下”，ReAct 的意义就会被说窄。论文里的 `Action` 不是推理完成之后的附属动作，而是推理过程中的一步。模型在某个 `Thought` 里发现信息不足，于是发起搜索或查找；工具返回 `Observation` 之后，下一段 `Thought` 再基于新信息调整方向。这里不是先有完整答案、再调用工具补引用，而是答案本身在工具反馈中逐步形成。

这个差别很关键。传统问答里，模型通常一次性读完输入，再生成答案；工具增强问答里，工具有时只是被当成检索上下文的前置步骤。ReAct 更接近一个交互式过程：模型可以先做一个局部判断，用行动验证这个判断，再把观察结果写回后续推理。行动不是打断推理，而是把推理推进到上下文之外。

论文里的 Wikipedia 任务很好说明这一点。模型可能先判断问题涉及某个人物，调用 `Search[...]` 找到页面；看到页面片段后，发现还需要另一个实体，于是继续 `Lookup[...]`；等命中的句子足够支持答案，再 `Finish[...]`。每一次动作都不是孤立工具调用，而是在回答“下一步还缺什么信息”。工具返回的结果，也不是简单拼接到 prompt 末尾，而是改变了模型对任务状态的判断。

放到今天的 Agent 工程里，这个结构仍然成立。代码 Agent 看到报错后先读相关文件，读完再决定改哪里，改完再跑测试；浏览器 Agent 看到页面状态后决定点击、输入还是等待；研究 Agent 搜索一轮之后根据结果质量决定是否换关键词或追原始来源。表面上这些系统的工具不同，但共同点是：工具调用承担了探索、验证和推进任务状态的职责。

这也解释了为什么工具设计不能只看“有没有这个能力”。同样是搜索工具，如果返回内容不可引用、不可定位、不可追溯，模型后续推理就很难稳定；同样是代码执行工具，如果输出被截断、错误码不清楚、工作目录混乱，模型就会基于残缺观察继续行动。ReAct 把工具放进推理链之后，工具返回的结构、粒度和可靠性都会直接影响 Agent 的判断质量。

所以，工具不是外挂，工具输出也不是装饰。对 ReAct 式 Agent 来说，工具调用就是推理过程的一部分：`Thought` 决定要问外部世界什么问题，`Action` 把问题发出去，`Observation` 再把外部世界的回答带回来。这个闭环一旦跑起来，模型才不只是“想得更长”，而是在不断把想法和现实对齐。

## Thought、Action、Observation 让过程变得可追踪

ReAct 轨迹的直接价值在于，它把一次 Agent 行为拆成几个可以定位的位置。传统问答只留下最终答案，答案错了，很难知道错误发生在理解问题、调用知识、组织推理还是生成结论的哪一步。CoT 虽然多留下了中间推理，但如果没有外部动作和观察，仍然只能看到模型在内部怎样自洽。ReAct 把 `Thought`、`Action`、`Observation` 放在同一条线上，失败就不再只是一个结果，而是一段可以回放的过程。

`Thought` 暴露的是模型当时的判断：它认为任务缺什么信息，下一步应该查什么，当前观察是否足够支持结论。`Action` 暴露的是模型实际选择的外部动作：它查了哪个实体，点了哪个页面，调用了哪个工具，运行了哪条命令。`Observation` 则暴露环境真实返回了什么：页面片段、工具输出、报错、测试结果、状态变化。三者连在一起，才能区分“模型没看见关键信息”和“模型看见了但读错了”。

这个区分对调试很重要。一个 Agent 提前 `Finish`，可能是因为 `Thought` 错误地判断信息已经足够；也可能是因为前面的 `Action` 查错对象；还可能是 `Observation` 本身太噪，模型从工具返回里抽错了句子。最终失败一样，但修法完全不同：前者要改任务分解或停止条件，第二种要改工具选择策略，第三种可能要改工具返回格式、截断策略或引用方式。

今天的 Agent runtime 和 eval transcript，也是在工程层面延续这个思路。一次浏览器任务失败时，只看最后截图往往不够；需要知道模型之前看到了什么 DOM、选择了什么按钮、点击后页面返回了什么状态。一次代码任务失败时，只看最终 patch 也不够；需要知道模型读过哪些文件、为什么改这段、运行了哪些测试、报错之后有没有误读。轨迹越完整，失败越容易从“模型不靠谱”拆成具体原因。

当然，轨迹本身不保证正确。模型可以写出看起来合理但实际错误的 `Thought`，也可以在 `Observation` 已经给出反证时继续坚持原方向。ReAct 的可追踪性更像是一种调试接口：它不会自动修好 Agent，但会把错误暴露在更具体的位置上。对工程系统来说，这已经很关键，因为只有错误被定位，后续才谈得上改 prompt、改工具、改环境反馈，或者把失败样本放进 eval。

## ReAct 对 Agent 工程的三个启发

把 ReAct 放回今天的 Agent 工程里，最值得保留的不是固定写成 `Thought`、`Action`、`Observation` 这几个词，而是它逼系统承认三件事：Agent 要留下过程，工具要服务推理，评测要能读懂失败。很多后来的工程实践看起来更复杂，本质上仍然绕着这三个问题转。

第一个启发是，Agent runtime 应该默认保留完整轨迹，而不是只保存最终回答。一次任务如果只有最后的自然语言输出，失败时就只能靠猜；如果能看到模型看过哪些信息、做过哪些动作、每次工具返回了什么，问题就可以被切开。生产系统里的 trace、run log、tool call history、browser recording、代码 patch 和测试输出，不只是审计材料，也是后续调试、评测和产品改进的输入。

第二个启发是，工具返回要服务后续推理。很多工具接口只追求“能调用”，但 ReAct 视角会继续追问：返回内容是否足够短，是否保留来源，是否能定位到具体页面、文件、行号或状态，是否把错误和空结果表达清楚。因为 `Observation` 会进入下一轮 `Thought`，所以工具输出越含糊，模型后续判断越容易偏。一个好工具不只是完成外部动作，还要把外部世界压缩成模型能继续推理的形状。

第三个启发是，eval 不能只看最终答案。最终 outcome 当然重要，任务有没有完成、状态有没有改变、测试有没有通过，这些都必须检查。但对 Agent 来说，只看 outcome 仍然不够。失败样本需要回到 transcript 里看路径：是前提判断错了，还是工具调用错了，是观察结果缺失，还是模型读到了反证却没有调整。只有把 outcome 和 transcript 连起来，eval 才能从打分工具变成排查工具。

这三个启发还能反过来约束 Agent 产品设计。如果一个系统没有轨迹，就很难让用户信任它；如果工具输出不适合被模型消费，再强的模型也会被坏观察带偏；如果评测只报一个通过率，团队就很难知道下一步该改 prompt、改工具、改任务拆分，还是改运行环境。ReAct 的简单之处在于 prompt 格式，真正留下来的工程问题并不简单。

## 这套范式没有解决什么

ReAct 容易给人一种错觉：只要把模型输出组织成 `Thought`、`Action`、`Observation`，Agent 就会自然可靠起来。但这套范式真正提供的是结构，不是保证。它让模型有机会交替推理和行动，也让过程更容易被检查；至于工具是否可靠、环境是否稳定、错误是否能恢复、权限是否安全，仍然要靠工程系统自己补上。

第一个没有自动解决的问题，是工具和环境本身的可靠性。`Observation` 来自外部世界，但外部世界可能返回空结果、过期内容、权限错误、网络超时、页面变化或不完整日志。如果模型把这些观察当成可靠事实继续推理，ReAct 轨迹反而会把错误带进下一步。换句话说，ReAct 可以让模型“看见”环境，但不能保证环境返回的是干净、完整、可用的信息。

第二个问题是轨迹成本。每一轮 `Thought`、`Action`、`Observation` 都会消耗上下文、时间和工具调用预算。简单任务如果也强行展开多轮轨迹，可能比直接回答更慢、更贵，也更容易引入无关观察。Google Research 的介绍里也提到，在需要大量动作的决策任务中，reasoning traces 不一定每一步都要出现，而是可以稀疏地放在关键位置。今天做 Agent runtime 时也类似：完整记录不等于无限展开，关键判断和关键状态要留下来。

第三个问题是错误恢复策略。ReAct 让异常更容易暴露，但它没有规定模型发现异常后应该怎样恢复。搜索不到结果时要不要换关键词，工具报错时要不要重试，浏览器卡住时要不要刷新，代码测试失败后要不要回滚修改，这些都是系统策略。没有这些策略，轨迹里只会出现一串“发现问题但继续乱走”的记录，调试时看得更清楚，不代表运行时就会自动变稳。

第四个问题是长期状态和权限边界。ReAct 的原始设定更像单次任务内的轨迹：在当前上下文里推理、行动、观察、完成。但今天的 Agent 往往会跨文件、跨会话、跨工具甚至跨真实账户行动。哪些记忆应该保留，哪些观察不能进入长期记忆，哪些工具动作必须人工确认，哪些权限不该交给模型，这些都不在 `Thought`、`Action`、`Observation` 三段式本身之内。

所以，ReAct 更适合被理解成 Agent 工程的底层骨架，而不是完整答案。它回答了一个关键问题：模型怎样把内部推理和外部行动接到同一条可观察轨迹里。但要把这条轨迹变成可靠产品，还需要工具设计、运行时隔离、错误恢复、权限控制和评测体系一起工作。它让 Agent 更容易被看见，也让失败更容易被定位；真正让 Agent 可靠的，是围绕这条轨迹搭起来的整套工程约束。
